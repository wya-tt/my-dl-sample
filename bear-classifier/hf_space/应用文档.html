Bear Classifier 熊类图像分类器 — 应用文档

在线访问

  • 应用链接：https://yaphets429-bear-classifier-test.hf.space/
  • 部署平台：Hugging Face Spaces（Gradio SDK）
  • 打开后即为网页交互界面，无需安装任何环境。
链接中的 __theme=system 表示跟随系统深/浅色主题；deep_link=... 是 Gradio 生成的分享深链参数，可省略，直接访问 https://yaphets429-bear-classifier-test.hf.space/ 也能使用。

一、目的

这是一个基于深度学习的图像分类 Web 应用，用于识别上传的图片属于以下三类熊中的哪一种：

类别含义
black黑熊
grizzly灰熊（棕熊）
teddy泰迪熊（玩具熊）

模型来自 fastai 课程《Practical Deep Learning for Coders》的经典示例，训练数据为通过图片爬虫收集的三类熊图片，使用迁移学习（ResNet）微调得到。本应用的目标是把训练好的模型部署成一个任何人都能在浏览器里直接使用的在线 demo。

二、使用方法

  1. 打开应用链接。
  2. 在左侧 "Upload a bear image" 区域上传一张图片（点击上传或拖拽，支持常见图片格式）。
    • 页面底部提供了 3 张示例图（黑熊 / 灰熊 / 泰迪熊），可直接点击试用。
  3. 模型会自动进行预测，右侧 "Prediction" 区域显示结果：给出最可能的类别及各类别的置信度概率（最多显示 3 类）。
  4. 想换图重测，重新上传即可。
首次访问若 Space 处于休眠状态，会有几十秒的唤醒/加载时间，属正常现象。

三、主要设计思路

1. 训练与推理分离

  • 训练在本地完成（见 02_production.ipynb），训练好的模型通过 learn.export() 导出为单个文件 export.pkl（约 45 MB），其中包含网络结构、权重、数据预处理流程与类别词表 vocab。
  • 推理/部署只需加载 export.pkl，无需训练代码与原始数据，应用启动即 load_learner("export.pkl")。

2. 用 Gradio 构建交互界面

  • 采用 gr.Interface 高层封装：输入为图片、输出为带概率的标签，几行代码即可生成完整网页 UI，无需手写前端。
  • 输出用 gr.Label(num_top_classes=3)，直观展示各类别置信度。

3. 选择 Hugging Face Spaces 而非 Binder 部署

  • 最初尝试用 Binder + Voilà 部署，但 Binder 会话空闲即回收、且运行子域名网络可达性差，多次出现 Binder inaccessible / 424 等问题，不适合长期稳定访问。
  • 改用 Hugging Face Spaces：免费、持久、单域名访问稳定，闲置只是休眠、访问即唤醒，更适合做长期可用的 demo。

4. 解决跨平台 / 跨版本的部署坑（关键工程细节）

模型在 Windows + Python 3.13 上导出，部署到 Linux 的 Space，踩了一系列坑并逐一解决：

  • WindowsPath 无法在 Linux 重建：在 app.py 加载模型前，将 WindowsPath 映射为 PosixPath（仅在 POSIX 系统上执行）。
  • pathlib._local 模块：Python 3.13 才有，因此 Space 也固定使用 Python 3.13（与导出环境一致）。
  • pydub 依赖 audioop：Python 3.13 移除了标准库 audioop，通过加入 backport 包 audioop-lts 解决。
  • pickle 反序列化所需的包：模型内部引用了 cloudpickle、fasttransform、plum、numpy(_core) 等，在 requirements.txt 中固定对应版本，避免反序列化失败。
  • 依赖版本一致性：fastai、fastcore、torch、numpy 等钉到与训练导出时相同的版本，确保模型能被正确加载。
  • Gradio 版本策略：直接使用最新版 gradio（6.x），由它自动拉取配套的 fastapi/starlette/pydantic/huggingface_hub，避免"旧版 gradio 撞上新版传递依赖"导致的接口不兼容问题。

Bear Classifier — Documentation

Live App

  • App link: https://yaphets429-bear-classifier-test.hf.space/
  • Platform: Hugging Face Spaces (Gradio SDK)
  • It opens directly as a web UI — no local setup required.
In the link, __theme=system follows the system light/dark theme; deep_link=... is a Gradio share deep-link parameter and is optional — visiting https://yaphets429-bear-classifier-test.hf.space/ directly works too.

1. Purpose

This is a deep-learning image classification web app that identifies which of the following three bear types an uploaded image belongs to:

ClassMeaning
blackBlack bear
grizzlyGrizzly / brown bear
teddyTeddy bear (toy)

The model comes from the classic example in the fastai course Practical Deep Learning for Coders. Its training data was collected via an image crawler for the three bear types and fine-tuned with transfer learning (ResNet). The goal of this app is to deploy the trained model as an online demo that anyone can use directly in a browser.

2. How to Use

  1. Open the app link.
  2. Upload an image in the left "Upload a bear image" area (click or drag & drop; common image formats supported).
    • Three example images (black / grizzly / teddy) are provided at the bottom of the page for quick testing.
  3. The model predicts automatically, and the "Prediction" area on the right shows the result: the most likely class and the confidence probabilities for each class (top 3).
  4. To test another image, just upload a new one.
On the first visit, if the Space is asleep, expect a few tens of seconds for wake-up/loading — this is normal.

3. Main Design Ideas

1. Separation of training and inference

  • Training is done locally (see 02_production.ipynb). The trained model is exported via learn.export() into a single file export.pkl (~45 MB), which contains the network architecture, weights, preprocessing pipeline, and the class vocabulary vocab.
  • Inference / deployment only needs to load export.pkl — no training code or raw data required. The app simply calls load_learner("export.pkl") at startup.

2. Building the UI with Gradio

  • Uses the high-level gr.Interface: image input, labeled-probability output — a complete web UI in just a few lines, no frontend code needed.
  • The output uses gr.Label(num_top_classes=3) to clearly show per-class confidence.

3. Choosing Hugging Face Spaces over Binder

  • The first attempt used Binder + Voilà, but Binder sessions are reclaimed when idle and its runtime sub-domain had poor network reachability, repeatedly causing Binder inaccessible / 424 errors — unsuitable for stable long-term access.
  • Switched to Hugging Face Spaces: free, persistent, single-domain and stable. When idle it merely sleeps and wakes on access — a better fit for a long-lived demo.

4. Solving cross-platform / cross-version deployment pitfalls (key engineering details)

The model was exported on Windows + Python 3.13 and deployed to a Linux Space, which surfaced a chain of issues, each resolved:

  • WindowsPath cannot be reconstructed on Linux: before loading the model in app.py, map WindowsPath to PosixPath (only on POSIX systems).
  • The pathlib._local module: exists only in Python 3.13, so the Space is also fixed to Python 3.13 (matching the export environment).
  • pydub depends on audioop: Python 3.13 removed the stdlib audioop, solved by adding the backport package audioop-lts.
  • Packages required for pickle deserialization: the model internally references cloudpickle, fasttransform, plum, numpy(_core), etc.; their versions are pinned in requirements.txt to avoid deserialization failures.
  • Dependency version consistency: fastai, fastcore, torch, numpy, etc. are pinned to the same versions used when exporting, so the model loads correctly.
  • Gradio version strategy: use the latest gradio (6.x) and let it pull matching fastapi/starlette/pydantic/huggingface_hub, avoiding interface incompatibilities from "old gradio meeting new transitive dependencies".
